# Research: AI model providers in the AURA monorepo

## Summary
Vertex AI/Gemini is the primary model provider across both apps for chat, embeddings, entity extraction, summarization, and KG generation. OpenRouter is the secondary/alternate provider exposed through the shared router and settings UI, while Deepgram is used only in AURA-NOTES-MANAGER for speech-to-text. Both apps keep legacy `vertex_ai_client.py` façades that now delegate to the shared router instead of calling provider SDKs directly.

## Findings
1. **Vertex AI/Gemini is the primary provider in AURA-CHAT** — `server/routers/chat.py` resolves the chat model as `request -> admin default -> env fallback`, and `backend/rag_engine.py` defaults to `config.RAG_MODEL_DEFAULT` with Gemini models listed in `backend/utils/config.py`. The same provider path is used by `backend/llm_entity_extractor.py` (entity/relationship extraction), `backend/llm_gatekeeper.py` (ambiguous-query reclassification), and `backend/utils/embeddings.py` (embeddings). The `backend/utils/vertex_ai_client.py` module is explicitly a legacy façade over `model_router`. (AURA-CHAT/server/routers/chat.py; AURA-CHAT/backend/rag_engine.py; AURA-CHAT/backend/utils/config.py; AURA-CHAT/backend/llm_entity_extractor.py; AURA-CHAT/backend/llm_gatekeeper.py; AURA-CHAT/backend/utils/embeddings.py; AURA-CHAT/backend/utils/vertex_ai_client.py)

2. **OpenRouter is the secondary/alternate provider in AURA-CHAT** — `backend/rag_engine.py` infers `provider="openrouter"` when a model name contains `/`, and `backend/utils/vertex_ai_client.py` forwards optional provider hints into the router. The admin settings router exposes model discovery and API-key validation for both `ProviderType.OPENROUTER` and `ProviderType.VERTEX_AI`, which means OpenRouter is supported at runtime but not the default path. (AURA-CHAT/backend/rag_engine.py; AURA-CHAT/backend/utils/vertex_ai_client.py; AURA-CHAT/server/routers/settings.py)

3. **AURA-NOTES-MANAGER uses Vertex AI/Gemini as the main LLM provider for KG processing and summarization** — `api/kg_processor.py` documents a Vertex AI client for embeddings and entity extraction, `services/summary_service.py` uses `LLM_SUMMARIZATION_MODEL` plus an admin override from `SettingsStore`, and `services/embeddings.py` routes embeddings through `model_router` using the configured provider. The provider/credential defaults are in `api/config.py` (`VERTEX_PROJECT`, `VERTEX_LOCATION`, `VERTEX_CREDENTIALS`, `GOOGLE_APPLICATION_CREDENTIALS`, `LLM_ENTITY_EXTRACTION_MODEL`, `LLM_SUMMARIZATION_MODEL`, `EMBEDDING_MODEL`). (AURA-NOTES-MANAGER/api/kg_processor.py; AURA-NOTES-MANAGER/services/summary_service.py; AURA-NOTES-MANAGER/services/embeddings.py; AURA-NOTES-MANAGER/api/config.py)

4. **AURA-NOTES-MANAGER also exposes OpenRouter and Ollama in its settings layer, but only as configurable providers** — `api/routers/settings.py` lists `ProviderType.OPENROUTER`, `ProviderType.VERTEX_AI`, and `ProviderType.OLLAMA`, and provides model discovery plus key storage/validation for those providers. This makes OpenRouter a supported alternative path rather than the hardcoded default. (AURA-NOTES-MANAGER/api/routers/settings.py)

5. **Deepgram is the only non-LLM provider found, and it is primary for audio transcription in AURA-NOTES-MANAGER** — `services/stt.py` calls Deepgram Nova-3 directly and requires `DEEPGRAM_API_KEY`. There is no fallback provider shown in that path. (AURA-NOTES-MANAGER/services/stt.py)

6. **Legacy compatibility layers remain in both apps** — `backend/utils/vertex_ai_client.py` (AURA-CHAT) and `services/vertex_ai_client.py` (AURA-NOTES-MANAGER) are both marked as legacy façades and preserve historical imports while delegating to the shared router. In NOTES, `USE_MODEL_ROUTER=true` can switch the façade toward the shared router-backed compat path. (AURA-CHAT/backend/utils/vertex_ai_client.py; AURA-NOTES-MANAGER/services/vertex_ai_client.py)

## Gaps
I could not inspect the internal `model_router` package itself, so the exact default-provider mappings inside `SettingsStore`/`ProviderType` are inferred from the app callers and settings routers rather than verified at source. Ollama is exposed in the settings API, but I did not find an app-specific runtime path that actively selects it.
