# Milestone v1.2: Settings Wiring E2E

**Status:** SHIPPED 2026-03-23
**Phases:** 14-17
**Total Plans:** 11

## Overview

AURA v1.2 wires SettingsStore end-to-end so every AI feature in both applications respects per-use-case model/provider configuration. Before this milestone, only chat in AURA-CHAT was fully wired — the remaining 7 consumers either bypassed SettingsStore with hardcoded env vars, read the `model` field while ignoring `provider`, or were blocked by an incomplete `ALLOWED_USE_CASES` allowlist.

This milestone introduces a shared `resolve_use_case_config()` utility, expands the use-case allowlist, fixes caching pitfalls, wires all 8 consumers across both apps, and validates the full settings→behavior flow with E2E tests.

## Phases

**Goal:** All use cases are configurable via the settings API, and a shared config resolution utility provides `{provider, model}` to every consumer with a reliable fallback chain
**Depends on:** Nothing (first phase of v1.2; v1.1 complete)
**Requirements:** API-01, API-02, FB-01, FB-02
**Plans:** 2 plans

Plans:
- [x] 14-01-PLAN.md — Sentinel cache fix + resolve_use_case_config() utility + tests
- [x] 14-02-PLAN.md — ALLOWED_USE_CASES expansion in both routers + router tests

**Success Criteria (all verified):**
1. PUT `/api/v1/settings/defaults/gatekeeper` returns 200 instead of 400
2. PUT `/api/v1/settings/defaults/relationship_extraction` returns 200 in both routers
3. `resolve_use_case_config("gatekeeper")` returns `{provider, model}` from SettingsStore with graceful fallback
4. Redis recovery resumes SettingsStore values within 30 seconds (no zombie-None cache)

**Details:**
- `resolve_use_case_config()` implemented with 3-step resolution: SettingsStore → env var → hardcoded default

- `ALLOWED_USE_CASES` expanded to include `gatekeeper` and `relationship_extraction` in both AURA-CHAT and AURA-NOTES-MANAGER settings routers
- 17 unit tests in `shared/model_router/tests/test_settings_store.py`

### Phase 15: Wire AURA-CHAT Consumers

**Goal:** Every LLM call in AURA-CHAT reads provider and model from SettingsStore and passes them explicitly to ModelRouter — no more hardcoded env vars or `/` heuristic provider guessing
**Depends on:** Phase 14
**Requirements:** PP-01, PP-02, PP-03, PP-04
**Plans:** 3 plans

Plans:
- [x] 15-01-PLAN.md — Shared infrastructure: provider propagation in VertexCompatModel + get_model() + OpenRouter JSON mode (Wave 1)
- [x] 15-02-PLAN.md — Wire entity extraction + relationship extraction to resolve_use_case_config() (Wave 2)
- [x] 15-03-PLAN.md — Wire gatekeeper + fix embeddings VERTEX_PROJECT check (Wave 2)

**Success Criteria (all verified):**
1. Entity extractor uses provider from SettingsStore for `entity_extraction` — not hardcoded Vertex AI
2. Gatekeeper routes through ModelRouter with configured provider — OpenRouter blanket skip removed
3. Embeddings calls `router.embed(provider=cfg["provider"])` — VERTEX_PROJECT check only fires for vertex_ai
4. Relationship extraction reads from SettingsStore independently, passes `{provider, model}` to ModelRouter
5. Changing provider takes effect on next LLM call without restart

**Details:**
- VertexCompatModel accepts optional `provider` parameter, propagates to GenerateRequest kwargs
- `get_model()` forwards provider through to VertexCompatModel constructor
- OpenRouter provider translates `response_mime_type: application/json` to `response_format: {type: json_object}`
- LLMEntityExtractor.__init__ calls `resolve_use_case_config("entity_extraction")` at init time
- Independent `_resolve_relationship_model()` for relationship extraction provider
- Gatekeeper: 30-line provider filtering block replaced with 12-line `resolve_use_case_config("gatekeeper")` call
- Embeddings: VERTEX_PROJECT RuntimeError wrapped with `_provider == "vertex_ai"` condition

## Milestone Summary

**Key Decisions:**
- Shared `resolve_use_case_config()` utility centralizes 3-step resolution chain (SettingsStore → env var → hardcoded default)
- Zombie-None cache fix with shorter error TTL (30s) before any consumer wiring
- `gatekeeper` and `relationship_extraction` added to ALLOWED_USE_CASES in both settings routers
- AURA-CHAT settings page uses existing axios instance instead of NOTES-MANAGER's fetchApi wrapper
- Settings route placed OUTSIDE RoleProtectedRoute wrapper — AdminSettingsRoute guard used instead
- `gatekeeper` and `relationship_extraction` mapped to 'generation' model type in frontend
- Use `_run_sync` from model_router.compat instead of `asyncio.get_event_loop().run_until_complete()` for Python 3.14 compatibility

**Issues Resolved:**
- 7 of 8 LLM consumers bypassed SettingsStore — all now read from it at runtime
- OpenRouter blanket skip in gatekeeper removed — JSON mode works per-provider
- VERTEX_PROJECT check in embeddings was provider-unaware — now conditional
- Summarizer had bare `except:pass` — replaced with logged error handling
- `ALLOWED_USE_CASES` was incomplete — expanded in both apps

**Technical Debt:**
- Dead code in `llm_entity_extractor.py` `_extract_partial_json` (lines 239-274 unreachable after `return {}`)
- `summarization` use case has no frontend picker UI (only configurable via API)
- VALIDATION.md files in Phases 14, 15, 17 remain in draft (nyquist_compliant: false)
