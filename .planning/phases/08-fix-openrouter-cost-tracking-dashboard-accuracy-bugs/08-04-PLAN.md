## Historical $0 Records — Decision

Per the research recommendation and anti-patterns guidance, this phase focuses on
**fixing forward-looking recording only**. Existing Redis records with `$0` costs
are not retroactively corrected. Rationale:

1. Historical records have estimated (not actual) token counts, so recalculating
   costs from them would still be inaccurate
2. Redis sorted set entries would need to be deserialized, recalculated, and
   re-serialized — complex migration for ephemeral data
3. The dashboard naturally ages out old data (30-day default window)

If retroactive correction is desired in the future, a standalone migration script
can iterate over Redis sorted set members, parse UsageRecord JSON, recalculate
costs using the now-populated pricing cache, and update the entries.
